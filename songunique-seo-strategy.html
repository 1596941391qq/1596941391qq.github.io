SEO Audit + Growth Strategy

SongUnique 不缺页面，缺的是能让人放心送出去的证据。

SongUnique 已经有完整的 PSEO 雏形：约 744 个 sitemap URL、8 个语言方向、recipient + occasion 场景页、基础 Product/FAQ/Breadcrumb schema。真正的瓶颈不在“有没有更多页面”，而在成交页、信任证据、样歌资产、站外背书和 AI 搜索可摘取结构。

Current Diagnosis

Site Type First

先按礼物站评估，而不是按 AI 工具站评估

用户买的不是一个音乐模型，而是一份能送给妈妈、伴侣、孩子、朋友或逝去亲人的情绪载体。SEO 评估标准必须围绕“是否可送、是否可信、是否不尴尬、是否能按时交付、是否可修改退款”。

用户真实担心

这首歌会不会尴尬？对方听完会不会有反应？我能不能把故事讲清楚？不满意能否退款？有没有样歌让我判断质量？

核心信任证据

样歌试听、歌词片段、真实评论、收件人反应、交付时间、退款规则、支付安全、多语言案例。

品牌主心骨

“AI song generator” 可以作为流量词，但不能做主叙事。主叙事应是 “personalized song gift”。

审计项执行展开

上面的 16 项不是标题清单，而是执行队列。SongUnique 的问题不在某一个单点错误，而在“成交入口、证据页、目录 hub、样歌资产、站外背书”没有被压成同一条增长链路。下面按可交付动作继续展开。

模块当前可确认状态增长影响具体动作
抓取入口robots.txt 存在；sitemap 当前为 XML；约 744 loc 和 5952 alternate。SongUnique 依赖大量 recipient、occasion、多语言页面，如果 sitemap 没被 Google 完整读取，页面矩阵会停留在“站内存在”，不能变成搜索入口。用 GSC 验证提交、发现、抓取、索引数量；若单 sitemap 识别不稳定，拆成 sitemap-index + 分语言 sitemap。
索引治理后台、支付、dashboard 等路径已有 robots 阻止；/track 仍是可访问 utility 页面。低价值页面进入抓取队列会稀释站点质量信号，也会让 Google 把订单工具页和礼物内容页混在一起评估。建立 index/noindex 白名单；/track 若不承担搜索流量，应 noindex；支付成功、取消、订单页同时做鉴权和 noindex。
canonical多数场景页 self canonical 正常；/create 缺 canonical。/create 是核心成交入口，缺 canonical 会削弱它作为主产品页的稳定性；多语言和默认英文路径也可能产生重复判断。/create 立即补 self canonical；抽查默认英文与 /en/；memorial、memory、funeral 等相近页面不得粗暴合并。
状态码12 个样本 0 失败，但未全量覆盖 744 URL。PSEO 站点最怕批量页面隐藏 404、soft 404 或跳转链，单看首页和几个核心页不能证明矩阵健康。从 sitemap 抽样 100-200 个 URL，检查 200、canonical、hreflang、跳转链；异常按模板页类型归因。
SERP 片段首页可用；/create 完全缺 title/meta；browse meta 偏短；多个场景页 title 可能截断。SongUnique 的点击理由不是“AI 很强”，而是“这份礼物适合我这个关系和场景”。片段不清楚会直接损失 CTR。优先重写 /create、browse、reviews、anniversary、mother、blessing；title 控制 50-60 字符，meta 写清收件人、场景、交付和安心点。
语义结构内容页 H1/H2 基本可用；/create 两个 H1 且无 H2。AI 搜索和 Google 都需要明确“页面要回答什么”。/create 当前像应用流程，不像可索引产品页。/create 单一 H1：Create a Personalized Song Gift；补 How it works、Choose recipient、Add story、Delivery/refund、FAQ。
页面角色首页、browse、reviews、create 都存在，但角色没有完全分开。页面角色不清会导致用户路径断裂：看完场景页不知道听样歌，看完评论不知道如何下单，看完目录不知道选哪个。首页做定位；browse 做决策；reviews 做证据；examples 做试听；create 做成交；compare 做竞品分流。
内容覆盖平均 947 词；browse 306 词，create 184 词。场景页有字数，不等于解决送礼焦虑。真正需要覆盖的是“写什么、避开什么、效果如何、能不能改、多久交付”。create 扩到 800-1200 词；browse 扩到 1000+；S 级页面补样歌、歌词方向、评论和 FAQ。
信息架构footer 链接很丰富，但 hub 层内容不足。大量平铺页面会让搜索引擎难以判断主次。礼物站需要从关系、场景、证据、成交形成清楚路径。首页 -> browse hub -> 分类 hub -> 具体页；reviews/examples 反向链接具体页；具体页统一导向 create。
内链分发全样本 726 内链；/create 0 内链。成交页没有内链会变成孤岛。即使用户能通过按钮进入，搜索引擎也缺少上下文信号。所有场景页首屏、样歌后、FAQ 后链接 create；create 底部链接 reviews、examples、pricing、browse。
Schema已有 Organization、WebSite、Product、FAQPage、BreadcrumbList；create 无 JSON-LD。结构化数据决定机器能否稳定读出产品、评分、价格、FAQ 和面包屑。/create 补 Product 或 SoftwareApplication、FAQPage、BreadcrumbList；reviews 补 Review/AggregateRating，但评分来源必须页面可见。
媒体 SEO多数页面图片 alt 可用；/create 4 张图 alt 全空。SongUnique 的核心证据是样歌和故事，不是纯文字。音频和图片如果没有文本结构，无法成为搜索资产。补 /create alt；建立 examples 页，每个样歌含收件人、场景、故事摘要、歌词片段、音频说明和 CTA。
CWV本次未跑 Lighthouse / CrUX。移动端礼物搜索通常时间紧，首屏慢、表单卡顿、音频加载慢都会影响创建率。跑首页、create、browse、reviews、anniversary、memorial；目标 LCP < 2.5s、INP < 200ms、CLS < 0.1。
转化信任有 4.9 rating、30+ countries、PayPal、Privacy、refund；脚本未识别 CTA。用户下单前最怕“生成出来不能送”。信任模块必须贴近 create，而不是散落在 footer。create 首屏放 Create + Listen to examples；支付前说明交付、修改、退款、隐私；reviews 按场景证明效果。
AI Search Ready有 FAQ 和 schema 基础，但缺事实页和对比页。AI 搜索会引用定义句、步骤、表格、FAQ。没有这些结构，SongUnique 很难进入“推荐 personalized song gift”的答案。新增 How SongUnique works、pricing、examples、reviews、SongUnique vs Songfinch/Songlorious/LoveTunesAI。
站外证据样本 external links 为 0，未形成第三方背书。礼物站只靠站内自述不够。外部礼物博客、婚礼资源、纪念资源和社区提及能补可信度。做 Personalized Song Gift Guide、Memorial Song Wording Guide、Wedding Song Gift Checklist，用 resource outreach、testimonial links、unlinked mentions 获取自然链接。

Four Lines

四线增长策略

第一线：导流捕获

不只打 AI song generator。优先 personalized song gift、custom song gift、song for mom、anniversary song、memorial song、Songfinch alternatives 这类礼物意图和比较意图。

第二线：主站内容系统

把首页、browse、reviews、examples、create 组合成礼物决策系统。用户从“我该送什么”一路走到“我怎么下单”。

第三线：技术底盘

先修成交页、索引白名单、sitemap/GSC、schema、hreflang、状态码和 CWV。技术修复不是装饰，是保护 PSEO 矩阵。

第四线：站外 SEO

用 gift guide、wedding checklist、memorial wording guide、examples library 做自然 outreach，不做垃圾目录堆链接。
